# Kerna Security Model & Threat Model

Kerna is designed as a **runtime trust layer** for agentic execution. It assumes that Large Language Models (LLMs) are highly capable but fundamentally untrusted actors prone to hallucinations, prompt injections, and unbounded execution loops.

## Core Assumptions

1. **LLMs are Untrusted**: Any input generated by an LLM is treated as potentially malicious or erroneous.
2. **Plugins are Untrusted**: External MCP plugins can be compromised or exhibit unexpected behavior.
3. **Execution Boundaries Must Be Enforced**: Operations that mutate state (file writes, network requests, process spawning) must cross a strict trust boundary.

## Trust Boundaries

- **LLM ↔ Kerna Core**: LLMs communicate with Kerna solely by requesting tool calls. They cannot directly execute code or mutate state.
- **Kerna Core ↔ MCP Plugins**: Plugins execute in isolated sandboxes or as separate processes over stdio. Kerna strictly controls their lifecycles.
- **Kerna Core ↔ OS**: Kerna acts as the gatekeeper to the host operating system, enforcing path restrictions, capability limits, and execution budgets.

## Mitigation Strategies

### 1. Budgets (The Circuit Breakers)
Every task is assigned a `BudgetConfig` that defines hard limits on execution:
- **`max_tool_calls`**: Prevents unbounded loops where an agent repeatedly calls tools without progress.
- **`max_llm_calls`**: Prevents excessive API usage and runaway costs.
- **`max_runtime_seconds`**: Hard timeout on the entire task.
- **`max_memory_writes`**: Limits durable memory poisoning.
- **`max_output_bytes`**: Prevents memory exhaustion from runaway stdout streams.

### 2. Capabilities and the Plugin Manifest
Plugins must define a `manifest.toml` that explicitly declares their capabilities (e.g., `fs_read`, `network_outbound`, `shell`).
- **Fail-Closed Default**: Without a manifest, a plugin is treated as high-risk and operates under "Legacy Warning" mode.
- **Least Privilege**: Plugins only receive the permissions they explicitly request and the user approves.

### 3. The Sacred Pipeline & Event Sinking
Every tool execution follows a strictly ordered pipeline:
1. `tool.call.requested`: Emitted when the LLM proposes a tool call.
2. `tool.policy.checked`: Emitted after policy and permission checks.
3. `budget.checked`: Emitted to ensure limits are not exceeded.
4. `tool.call.started`: Emitted right before execution.
5. `tool.call.completed` / `failed`: Emitted with the result.

This pipeline ensures 100% observability and non-repudiation. All events are synced to a durable SQLite store.

## Known Threat Vectors & Mitigations

| Threat | Mitigation |
|--------|------------|
| Prompt Injection | Separated execution planes; injected commands still hit budget and capability checks. |
| Tool Output Poisoning | LLMs process tool outputs, but Kerna enforces limits on output size (`max_output_bytes`) and limits follow-up actions. |
| Infinite Loops | Strictly bounded by `max_tool_calls` and `max_llm_calls`. |
| Unintended File Modification | Sandbox restrictions, allowed path enforcement, and explicit approval requirements. |
